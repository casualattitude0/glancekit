# 交易計畫 JSON 格式

Glancekit 的 Stocks glance 會監看你指定的一個 JSON 檔。檔案一存檔就自動重新載入，
不需要重開 app。

## 匯入與歷史

匯入和切換計畫都在 **glance 本身**（選單列 popover、工具視窗、Quick Switch），
不必進 Settings：

- **匯入** — 選一個 JSON 檔，之後就持續監看它。
- **記錄** — 載入先前的計畫。每一份載入過的計畫都會依 `date` 存一份到
  `~/Library/Application Support/Glancekit/stock-plans/`，保留最近 60 份。
- **即時** — 從歷史檢視回到目前監看的檔案。

檢視歷史計畫時**通知會暫停**：翻看上週四的關卡不該用今天的價格發通知。若你是因為
今天的計畫還沒產出、想真的用那份舊計畫，按 **設為使用中** 才會重新啟用通知。

計畫日期不是今天時，畫面上會標示「非今日」— 均線每天都在動，過期的計畫會拿舊數字
判斷而且不會知道自己錯了。

## 設計前提

`condition` 是寫給**人**看的散文。程式不會嘗試理解「回踩不破、量縮」這種句子。
分工是這樣的：

- **程式決定「什麼時候該打擾你」** — 從它讀得懂的數字判斷。
- **散文原封不動放進通知內容**，由你決定「該做什麼」。

所以你現有的 plan 檔**不用改就能用**。下面的 `trigger` 是可選的，加了會讓判斷更準。

## 頂層結構

```jsonc
{
  "date": "2026-07-20",              // 用來做每日去重；日期一換，已觸發紀錄全部重置
  "generatedFrom": "…",              // 純備註
  "market": {
    "structure": "bull",
    "shortTerm": "bearish",
    "verdict": "回檔佈局窗口，分批不搶",
    "gate": {                        // 大盤總開關，掛在加權指數上
      "ladder": [42668, 42358, 42268],   // 階梯；破「最後一道」才停新倉
      "source": "…",
      "rule": "…",
      "reclaimLine": { "price": 46910, "source": "…", "note": "…" }
    },
    "usWatch": { "note": "…" },      // 外部市場，只顯示不判斷（我們沒有那指報價）
    "events": [{ "type": "fomc", "date": "2026-07-28", "note": "…" }]
  },
  "plans": [ /* 每檔一個 */ ]
}
```

`gate.ladder` 是**階梯不是單線**：破第一道只是警告，破**最後一道**才停止所有
entry / add / reentry 的通知。每一道破了都各發一則通知。只讀第一道的話，計畫在第一次
晃動時就整個凍結，而計畫本身認為那是可以撐過去的。trim / reduce / cut 不受影響 ——
大盤轉弱時該跑的還是要跑。單線寫法 `"line": 42268` 仍支援。

## 每檔的結構

```jsonc
{
  "stockId": "TPEX-3491",            // 見下方「代號寫法」
  "name": "昇達科",
  "flag": "warming",                 // 純顯示
  "score": 3,                        // 純顯示
  "evidence": [{ "item": "…", "value": "…" }],
  "levels": { "entry": {…}, "add": {…}, "trim": {…}, "cut": {…}, "reentry": {…} },
  "events": [{ "type": "ex_dividend", "date": "2026-07-23", "value": 6.8, "note": "…" }],
  "watchToday": ["…"],
  "review": null
}
```

### 代號寫法

`stockId` 接受：`TWSE-2330`、`TPEX-3491`、`2330.TW`、`3491.TWO`、裸碼 `2330`（視為上市）。
大盤用 `TAIEX`。**上櫃股票請務必寫 `TPEX-` 或 `.TWO`** — 裸碼會被當成上市而查不到報價。

## Level

```jsonc
"entry": {
  "price": 1355,                     // 數字，或區間陣列 [1348, 1355]
  "condition": "收盤突破且量≥2602張",  // 散文，原樣進通知
  "size": { "action": "+1/3", "after": "1/3" },
  "source": "平台頸線",
  "trigger": { /* 可選，見下 */ }
}
```

### 沒有 `trigger` 時的預設推論

| 項目 | 推論方式 |
|---|---|
| 方向 | `entry`/`add` → 向上突破；`trim`/`cut` → 向下跌破 |
| 比較價 | `price`（陣列則取下緣） |
| 收盤確認 | `condition` 裡有「收盤」兩字 → 只在收盤前後判斷 |
| 量能門檻 | 從 `condition` 抓 `量≥NNNN張` 這個句型 |
| `reentry` | **不會自動觸發** — 句型太複雜，只顯示不發通知。要它發通知就得寫 `trigger` |

### `size` — 要執行多少

```jsonc
"size": { "action": "+1/3", "after": "1/3" }
```

- `action` — **要執行的動作**，例如 `+1/3`、`sell 1/2 of holdings`、`sell all`。
  這會出現在**通知標題**，因為橫幅可能只顯示標題 —— 「進場 1355」沒告訴你要買多少，
  而那正是這個工具存在的理由。
- `after` — 執行後的部位，例如 `1/3`、`0`、`原水位`。

`action` **原文照搬，不翻譯**。把交易指令翻錯，比不翻譯糟得多。

舊格式 `"size": "1/3"`（純字串）仍可讀，會當成 `after`。

### 關卡種類

`entry` → `add` → `trim`（釋出）→ `reduce`（減倉）→ `cut`（出清）→ `reentry`，
畫面依此順序顯示。

### `positionPath` / `target`

```jsonc
"positionPath": { "max": "2/3", "up": ["0","1/3","2/3"], "down": ["2/3","1/3","1/6","0"], "note": "…" },
"target": { "band": [1450, 1470], "source": "…" }
```

`positionPath.max` 會附在通知裡（「上限 2/3」），`note` 也會。`target` 只顯示，不發通知。

## `trigger`（可選，精確控制）

```jsonc
"trigger": {
  "op": "crossAbove",          // 見下表
  "price": 1355,               // 省略則用 level 的 price
  "band": [1300, 1355],        // enterBand 用
  "ref": "literal",            // literal（預設）| ma20 | ma60
  "onClose": true,             // 只在收盤(13:25–13:30 及收盤後確認)判斷
  "volumeLots": { "min": 2602 },
  "confirmWithinDays": 3,      // 多日 op 用，預設 3
  "approachBands": [10, 5, 2]  // 此關卡專用的接近提醒級距；[] = 關閉
}
```

### `op`

| 值 | 意思 |
|---|---|
| `crossAbove` | 由下往上穿過線 |
| `crossBelow` | 由上往下跌破線 |
| `enterBand` | 從區間外進入 `band` 區間（回踩承接） |
| `reclaimWithinDays` | 跌破後 N 個交易日內收盤收回 →「破1195後3日內收回」 |
| `breakdownHeldDays` | 跌破後連續 N 個交易日都沒收回 →「跌破後3日未收回」 |

### `ref`

`ma5` / `ma10` / `ma20` / `ma40` / `ma60` / `ma120` / `ma240`。

**用 `ref` 時必須同時附數字 `price`**（本次計算的均線值）。app 以那個數字為準、不重算
均線 —— 這樣警報用的數字跟你早上讀的報告是同一個，不會出現兩邊對不起來的情況。
`price` 缺席時才會退而用 app 自己算的均線。

不認得的 `ref` 值會退回 `literal`，**不會讓整份計畫解析失敗** —— 新增一個 app 沒聽過
的均線，不該讓當天所有警報一起消失。

### `volumeLots`

單位一律是**張**。

| 欄位 | 意思 |
|---|---|
| `min` / `max` | 絕對張數下限／上限（`max` 就是「量縮」的寫法） |
| `multiple` | N 日均量的倍數下限，例如 `1.5` |
| `maxMultiple` | N 日均量的倍數上限，例如 `0.8` 表示量縮 |
| `refAvgDays` | 均量取幾日，預設 20 |

**相對量能需要日線資料。** 用了 `multiple` / `maxMultiple` 的關卡，在日線抓到之前
**無法判斷、因此不會觸發** —— 這種關卡在畫面上會標示 **待日線**，不會假裝它已就緒。
日線每天在第一次更新時就會抓（不再等收盤），收盤後再抓一次補當日 K。

`"volumeLots": { "multiple": 1.5 }` 比 `{ "min": 2602 }` 好 — 前者每天自己重算，
後者是 07-20 那天的快照。

## 接近提醒

兩種單位，不會混用：

| 來源 | 單位 | 用途 |
|---|---|---|
| `trigger.approachBands`（計畫指定） | **元** | 例如 `[10, 5, 2]` = 距離 10/5/2 元。建議至少給 `entry` 與 `cut` |
| Settings 的全域設定（預設 10/5/2） | **%** | 沒寫 `approachBands` 的關卡才會用到 |

通知會明講是哪一種（「接近停損 2 元」vs「接近停損 2%」），兩者不可能被看錯。

- **有方向性。** 距離只算「還能走到」的那一側 — 進場算價格在線**下方**多遠，停損算
  在線**上方**多遠。在進場線上方 5% 不叫接近，那叫已經穿過去了；不分方向的話，每一個
  已經被拋在後面的關卡都會發提醒。
- **一次只發最近的一級。** 從 12% 直接跳到 1.5%，只會收到一則「2%」，不會 10%、5%、
  2% 三則一起來。加上每天每級只發一次，價格在邊界附近震盪也不會洗版。
- **不受 `onClose` 限制。** 「距離停損 2%」的價值就在於盤中還來得及反應 —— 而那正是
  收盤條件本身安靜著的時段。通知內文會註明該關卡需收盤確認。
- 大盤破線時，entry/add/reentry 的接近提醒**也一併停止**，跟正式觸發一致。
- 在 Smart Panel 上，接近提醒永遠排在真正觸發之後。

每個關卡可用 `trigger.approachBands` 單獨覆寫（緊的停損可能想更早知道）。

## 持股 holding.json

另外匯入一個持股檔（glance 上的「持股 → 匯入」），關卡的部位比例就會直接換算成**股數**。

```jsonc
{
  "updatedAt": "2026-07-20",     // 非今日會標示，過期股數比沒有更危險
  "cash": 50000,                 // TWD，用來標示買不買得起
  "positions": [
    { "stockId": "TWSE-2330", "name": "台積電", "shares": 36, "avgCost": 2402 }
  ]
}
```

`shares` 是**股**不是張。`stockId` 寫法同計畫。存檔後自動重載。

### 股數怎麼算出來的

**賣出是精確的**，因為計畫把賣出寫成相對持股的：

| `size.action` | 持有 36 股 → |
|---|---|
| `sell 1/2 of holdings` | 18 股 |
| `sell all` | 36 股 |

無條件捨去 —— 少賣還有部位，多賣賣不出來。

**買進要推算滿倉**，而計畫自己就給得出來：一個步驟同時寫了做什麼和做完在哪，
`{"action": "+1/3", "after": "2/3"}` ⇒ 你現在必須站在 `2/3 − 1/3 = 1/3`。
所以一單位就等於你目前的持股，「+1/3」就是**再買一次目前的量**。不用設定、不用猜，
是計畫自己的算術加上真實部位推出來的。

推不出來的情況會直接說，不會給你一個來歷不明的數字：

| 情況 | 顯示 |
|---|---|
| `entry`（0 → 1/3）且未持有 | 建倉第一筆，無基準可推算 |
| `entry` 但已持有 | 已持有，此步視為已完成 |
| 該檔完全沒持股 | 未持有 |
| `buy back to prior level` | 回補至原水位 |

### 現金

買進量**照計畫算，不會被現金砍掉**，但現金不夠時會加註「現金僅夠 N 股」。
計畫想要的量跟你實際買得起的量都看得到。

### 直接編輯

持股區塊的「編輯」可以改股數、均價、現金，也能新增／刪除持股。儲存會**寫回同一個
持股檔**。

- **之後從交易紀錄重新匯出會覆蓋這裡的修改。** 這個優先順序是對的（紀錄才是帳），
  但意思是 app 裡的編輯屬於「盤中修正」，不是永久紀錄。
- 儲存只改動它認得的欄位，**其他欄位原樣保留** —— 你之後在 holding.json 加的任何
  東西不會被默默吃掉。
- 檔案會被重新排版成 pretty-print + 鍵值排序（diff 穩定，但第一次儲存後格式會變）。
- 儲存後 `updatedAt` 自動設為今天，股數換算立刻跟著新部位重算。

### 沒在計畫裡的持股

畫面會標示「無計畫」。那些部位**沒有任何停損或通知** —— 這是唯一一種會一邊看起來
乖乖的、一邊傷害你的東西。

> 所有股數都是依**匯入當下的持股檔**計算。減碼後記得重新匯出，否則下一個關卡還是拿
> 舊部位在算。

## 通知行為

- **每個 level 每天最多發一次**（接近提醒是每級距每天一次）。`date` 換日才重置。
- 載入計畫後的**第一個 tick 只記錄現價、不發通知** — 否則一開 app 就會被目前價格
  已在線下方的所有 cut 通知洗版。
- `onClose` 的條件盤中不發，只在 13:25–13:30 判斷，收盤後再確認一次。
- 通知內容 = 目前價 + `condition` 原文 + `size` + `source`。
